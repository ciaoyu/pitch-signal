## 一、Sprint 目标

1. **建立可信数据基础**：打通球员/球队/赛程/赔率的真实数据链路，消除 mock 假数据。
2. **补齐关键功能缺口**：完成前端差异化功能（SVG 对位阵型图、角球追踪条、预测 Tab）、数据可信度标记。
3. **偿还技术债务**：拆分单体文件实现模块化架构，统一 API 路由与前端调用层。

## 五、风险 & 缓解措施

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| **ESPN API 无公开免费接口** | 高 | P0 阻塞 | 提前调研 ESPN Core API / 第三方聚合 API（SportMonks）；若全部收费，降级为手动维护静态 roster JSON + 定期更新脚本 |
| **盘口替代数据源同样不可用** | 中高 | P1 阻塞 | Day 4 同步调研 3 个候选数据源；若均不可用，立即实施优雅降级方案（显示"实时赔率暂未接入"，隐藏假数据），不消耗额外时间 |
| **拆分 server.js 引入行为回归 bug** | 中 | P1 阻塞 | 拆分前先写 API 快照测试（保存所有 35 个端点的预期响应）；拆分后逐一路由对比，增量重构而非大爆炸 |
| **预测引擎输入层真实化后输出仍不可信** | 高 | P2 降级 | 明确预期：Sprint 目标不是"预测准确"而是"输入真实 + 输出标注"。若短时间内无法训练可信模型，P2 交付标准为：输入数据真实 + 输出带置信度标注 + 免责声明 |
| **3393 行 index.html 重构引入 UI 回归** | 中 | P1 阻塞 | 前端 API 调用层统一时采用"非侵入式"策略，先抽取调用逻辑，不重构 DOM 结构 |
| **2 周时间不足以完成全部任务** | 中 | 整体延期 | 严格按 P0→P1→P2→P3 顺序推进；Buffer Day 为兜底；P2/P3 任务可延至下个 Sprint |

## 七、一句话建议

> **先修数据，再修代码**——当前最大的用户体验灾难不是架构臃肿，而是满屏假数据给用户造成"这个产品不可信"的印象；Day 1-5 的优先级压倒性地高于 Day 6-10，宁愿第二个周末补架构重构，也不要在假数据地基上盖漂亮房子。
