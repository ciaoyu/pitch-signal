### 2.3 Mock 数据污染分析

**标记为 ✅ "已完成"但实际返回 mock 数据的端点**：

| 端点 | Mock 程度 | 影响 |
|------|----------|------|
| `/api/player/:id/enhanced` | **100% mock** | recentForm/clubStats/nationalStats 全随机 |
| `/api/team/:id/enhanced` | **80% mock** | 进攻/防守等指标均为 ratings 平均值换算 |
| `/api/match/:id/head-to-head` | **100% mock** | 历史交锋全部随机生成 |
| `/api/match/:id/news` | **100% mock**（无 Tavily key 时） | 新闻标题和内容均虚构 |
| `/api/odds/:matchId` | **100% mock**（无 Odds API key 时） | 盘口数据由 ratings 估算 |
| `/api/predict/:matchId` | **间接 80% mock** | 输入数据为 mock，输出自然不可靠 |

**核心问题**: 需求文档将"有端点返回数据"等同于"功能已完成"，不论数据是否真实。这在产品层面是重大缺陷——用户看到的"近10场场均进球 1.8"、"大战历史 12 次参赛"、"盘口隐含概率 45%"均无事实依据。

## 三、风险点

### 3.1 数据源风险

| 风险 | 等级 | 描述 |
|------|------|------|
| **ESPN API 不稳定** | 🔴 高 | 无官方文档，完全逆向工程。接口字段名、路径可能随时变更。`/teams/:id/roster` 当前返回空数组，核心数据源缺失。 |
| **The Odds API 不可用** | 🔴 高 | 免费层 500 req/月，且持续返回 `no_match_found`。World Cup 赛事 ID 映射可能根本不对。若不付费升级或换源，盘口功能名存实亡。 |
| **Tavily API 缺 Key** | 🟡 中 | 新闻功能完全依赖 mock。若 Tavily key 未配置或不支持中文/足球领域搜索，新闻价值为零。 |
| **OpenWeatherMap 缺 Key** | 🟡 中 | 天气数据无法获取，影响场地影响分析。但世界杯场馆多在室内/可关闭顶棚，影响有限。 |
| **无 xG/xA 真实数据源** | 🟡 中 | 文档多次引用 FBref/Opta 作为 xG 数据源，但实际均未接入。所有 "xG=1.65" 类数据均为 mock。 |
| **ratings.json 评分主观** | 🟢 低 | 748 人评分为手工/半自动生成，无实时更新机制。球员状态变化、转会、伤病不会反映在评分中。 |

### 3.2 架构风险

| 风险 | 等级 | 描述 |
|------|------|------|
| **零数据库** | 🟡 中 | 全部使用内存缓存 + 本地 JSON。服务重启丢失所有缓存数据（盘口快照、预测缓存）。预测历史无法持久化。 |
| **单进程无容错** | 🟡 中 | 纯 Node.js http 模块，无进程管理。ESPN 一次超时可能导致整个请求队列阻塞。无 health check。 |
| **无 API 限流** | 🟢 低 | ESPN API 无认证，但高频请求可能被 IP 封禁。当前无请求节流机制。 |
| **预测引擎冷启动** | 🟢 低 | 所有模拟数据无持久化，每次重启 Monte Carlo 重新运行 10000 次，浪费 CPU。 |

### 3.3 依赖风险

| 依赖 | 类型 | 风险 |
|------|------|------|
| `lib/prediction.js` | 自研 | 代码质量未知，Elot指导 Poisson 的逻辑未经回测验证 |
| `lib/elo.js` | 自研 | 需验证 K-factor 和初始 rating 配置合理性 |
| `lib/poisson.js` | 自研 | λ 参数对 mock 数据敏感，真数据下预测质量未知 |
| `lib/qualification.js` | 自研 | Monte Carlo 10000 次模拟，依赖 mock 输入 |
| `lib/matchReview.js` | 自研 | AI vs Actual 偏差分析，含 LLM 调用注释但未接入 |
| Chart.js | CDN | 前端依赖，离线模式不可用（Service Worker 未缓存） |
| Tailwind CSS | CDN | 同上 |

## 四、建议

### 4.1 接下来 2 周最该做的 3 件事

#### 🥇 第一优先：修复真实数据链路（打通任督二脉）

**目标**: 让 "✅ 已完成" 真正意味着"返回真实数据"。

1. **ESPN roster 问题排查** — 这是球员/球队数据的根基。验证 API 端点是否变更，或改用 ESPN 的 `/athletes` 端点 + teams 关联重建 roster。
2. **The Odds API 替代方案** — 要么付费升级 The Odds API（$99/月），要么切换到 Football-data.org（免费 10 req/min）或 API-Football（付费但数据全面）。
3. **接通至少一个真实统计源** — 优先接通 ESPN match log 获取球员近 N 场真实数据，替代当前所有 enhanced 端点的 mock 返回值。

> **不做这件事的后果**: 整个预测系统建立在沙子上。即使前端做得再漂亮，用户很快会发现数据是假的，信任归零。

#### 🥈 第二优先：完成前端差异化功能（让用户看得见价值）

1. **对位阵型图 SVG 渲染** — 后端已完整输出坐标和连线数据，前端只需画 SVG。这是最大视觉差异化点，可直接用于社交媒体传播。
2. **角球实时追踪条** — 后端已输出 progress.actual/expected 和 pace，前端只需一个进度条组件。
3. **预测 Tab 前端实现** — `/api/predict/:matchId` 已就绪，前端展示胜率、比分分布、相似比赛参考即可。

> **投入产出比**: 3 个前端功能均已有完整后端支撑，纯前端工作量，单人可以 5-7 天完成。

#### 🥉 第三优先：建立数据可信度标记系统

当前最大产品风险不是功能缺失，而是**用户无法区分真实数据和 mock 数据**。在每一个返回 mock 数据的 UI 元素上加上 `⚠️ 模拟数据` 标记，并在 API 响应中增加 `dataQuality` 字段：

```json
{
  "data": { ... },
  "_meta": {
    "dataQuality": "mock" | "estimated" | "live",
    "source": "ratings.json" | "espn_api" | "odds_api",
    "confidence": "high" | "medium" | "low"
  }
}
```

这比实现新功能更重要——**诚实比好看更值钱**。

### 4.2 哪些功能可以先砍掉（或降级）

| 功能 | 建议 | 理由 |
|------|------|------|
| **AI 异动解读** | ✂️ 砍掉 | 需要 LLM API + 盘口真数据，两个前提都不具备。v2 再说。 |
| **盘口变化原因推测** | ✂️ 砍掉 | 同上。在盘口真数据能跑通前，异动检测本身就是空的。 |
| **新闻重要性分级 🔴🟡🟢** | ⬇️ 降级 | Mock 新闻做分级没意义。等 Tavily 真实数据接通后再做。 |
| **比赛环境分析 /api/match/:id/context** | ⬇️ 降级 | 场馆+天气的两个独立端点已存在，一站式聚合端点属于 nice-to-have。 |
| **ML 模型优化 / 多维特征输入** | ✂️ 砍掉 v1 | 在真实数据链路不通的情况下训练 ML 模型是 Garbage In Garbage Out。先把基础数据跑通。 |
| **淘汰赛升级盘口 API** | ✂️ 推迟 | 小组赛数据都还没跑通。 |
| **预测历史 / 准确率** | ⬇️ 降级 | 需要数据库支撑。先做内存缓存版（只显示当前 session 内的预测记录）。 |
| **出线形势数学分析** | ⬇️ 保留但降噪 | Monte Carlo 引擎已有，但 10000 次模拟的结果不要作为"权威数据"展示——加上置信区间和水印。 |
