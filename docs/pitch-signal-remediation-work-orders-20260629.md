# 2. 工作包 A：前端核心修复

## A1：修复赛程页空白

### A1.1 当前问题

复现路径：

1. 打开网站，默认进入“比分 Live”。
2. 等待 Live 页加载当天比赛。
3. 点击“赛程 Schedule”。
4. 页面只显示日期区域左右箭头。
5. 日期按钮、日期标题和比赛列表全部为空。

后端 `/api/schedule` 实际返回正常赛程，因此不是后端没有数据。

### A1.2 已确认根因

`static/js/app.js` 当前通过以下条件判断是否加载完整赛程：

```js
if (state.tab === 'schedule' && !state.scheduleCache.length) {
    loadSchedule();
}
```

Live 页加载后会先把当天比赛写入 `state.scheduleCache`：

```text
Live 写入一场比赛
→ scheduleCache.length > 0
→ 进入赛程页时不调用 loadSchedule()
→ 日期栏和完整赛程从未渲染
```

根本问题是同一个 `scheduleCache` 同时承担：

- Live 临时比赛缓存。
- 完整赛程是否已加载的状态标志。

这两个概念不能继续混用。

### A1.3 修改目标

- 首次进入赛程页时必须加载并渲染完整 `/api/schedule`。
- Live 页仍可预填当天比赛，不能破坏比赛详情对赛程缓存的依赖。
- 完整赛程加载状态必须独立表达。
- 同一完整赛程请求不能无意义重复执行。
- 请求失败后必须允许重试。
- 页面不能出现无解释空白。

### A1.4 允许修改的文件

- `static/js/state.js`
- `static/js/app.js`
- `static/js/schedule.js`
- 自动生成的 `static/js/bundle.js`
- 构建脚本自动更新的 `templates/index.html` bundle 版本号

### A1.5 建议实现

在共享 State 中增加明确状态，例如：

```js
scheduleLoaded: false
scheduleLoading: false
```

也可以使用共享的 `scheduleLoadPromise`，但必须达到以下行为：

```text
未加载且未在加载
→ 发起完整赛程请求

正在加载
→ 复用当前请求，不重复请求

已加载
→ 使用完整缓存渲染

加载失败
→ 清除 loading 状态
→ loaded 保持 false
→ 展示错误
→ 下次允许重试
```

完整赛程加载成功后，必须用完整结果替换或规范合并临时缓存，不能保留重复比赛。

### A1.6 修改步骤

1. 保存工作前 `git status --short`。
2. 启动当前服务。
3. 按“Live → Schedule”复现空白。
4. 记录 `/api/schedule` 状态和浏览器控制台。
5. 检查 State 中所有赛程相关字段和使用方。
6. 增加独立的完整赛程加载状态。
7. 修改 `switchTab()` 的赛程加载条件。
8. 调整 `loadSchedule()` 的成功、失败、重复调用和重试状态。
9. 检查赛程缓存合并是否可能产生重复比赛。
10. 执行 `npm run build:js`。
11. 刷新实际运行页面；必要时清理旧 Service Worker/站点缓存。
12. 按全部验收场景验证。
13. 运行完整测试和 Git 检查。
14. 独立提交 A1。

当前隐藏的旧版：

- `#ai-bot-root`
- `static/js/bot.js`
- `static/css/bot.css`

旧版入口虽然被 CSS 隐藏，但脚本仍在加载和执行，并产生：

1. 多轮对话历史。
2. 当前页面和比赛上下文。
3. “AI 问答”和“留言 / 意见”双模式。
4. 留言入库。
5. 发送中、成功和失败状态。
6. 中英文同步。

### A2.3 修改目标

最终页面只存在一个 AI 浮动入口和一个 AI 面板。

用户应能：

- 点击快捷问题。
- 手动输入并发送。
- 连续追问并保留对话历史。
- 在比赛详情中询问当前比赛。
- 切换到留言模式并真实保存。
- 获得准确的发送成功、失败或功能未配置提示。
- 在中英文间正确切换。

# 3. 工作包 B：修复轮次和组别标签

```text
预测卡：第 ?
积分榜：MD ?/3
```

这不是单纯视觉问题，而是前端依赖的 `matchday` 或 `group` 字段没有稳定返回。

## B.2 修改目标

- 小组赛显示正确组别和轮次。
- 淘汰赛显示正确阶段，如 R32、R16、QF、SF、Final。
- 数据无法确认时显示明确降级文案，不显示裸 `?`。
- 不推测或伪造轮次。
- API 与前端对字段含义保持一致。

## B.3 首要检查步骤

1. 保存 Git 状态。
2. 抓取 `/api/schedule` 和 `/api/standings` 当前样本。
3. 检查 ESPN 原始数据是否包含轮次或赛段。
4. 检查 `parseEvent()` 是否丢失字段。
5. 检查 `/api/standings` 是否能够提供可靠 matchday。
6. 检查淘汰赛是否误走小组赛标签模板。
7. 定义统一字段和降级规则。
8. 先补测试，再修改实现。
9. 浏览器抽查小组赛与淘汰赛。

## B.4 预计涉及文件

- `lib/parse-event.js`
- `lib/routes/core.js`
- `static/js/elo-prediction.js`
- `static/js/standings.js`

## B.5 边界和禁止事项

# 8. 总审核流程

各负责人交付后，由总审核人统一执行：

## 8.1 范围审核

- 是否只修改任务允许文件。
- 是否混入无关改动。
- 是否覆盖其他人的工作。
- 生成文件是否由正确命令产生。

## 8.2 代码审核

- 根因是否真正解决。
- 是否只是增加绕过逻辑。
- 错误状态是否诚实。
- 是否破坏 DOM、API、State 或缓存契约。

## 8.3 自动验证

```bash
npm run build:js
npm test
HEALTHCHECK_BASE_URL=http://127.0.0.1:5099 npm run healthcheck
git diff --check
```

## 8.4 浏览器验证

- 首页启动。
- 五个主 Tab。
- 赛程加载和日期切换。
- 比赛详情。
- 积分与淘汰赛。
- 球队详情。
- AI 面板。
- 中英文。
- 控制台错误。

## 8.5 审核结论

每个任务只给出以下三种结论：

- **通过**：所有完成定义满足，可以进入下一阶段。
- **需返工**：列出未满足的验收项和证据。
- **有条件通过**：核心问题解决，但有明确、可隔离且已记录的非阻塞遗留项。
