## 二、路线决策：「先修架构」vs「先修数据」

### 2.1 Sprint 规划师的「先修数据」建议是否正确？

**正确，且理由充分。**

Sprint 规划师的一句话建议：

> 先修数据，再修代码——当前最大的用户体验灾难不是架构臃肿，而是满屏假数据给用户造成"这个产品不可信"的印象。

这个判断的底层逻辑：

1. **用户视角**：用户打开页面，看到「德国近 10 场场均进球 1.8」「巴西小组出线概率 89%」「葡萄牙 vs 法国盘口隐含概率 45%/30%/25%」——这些数字都是 mock 随机生成的，**零信息量**。用户不需要知道 server.js 是 3154 行还是 150 行，但用户会在 30 秒内发现数据是假的。

2. **产品价值**：PitchSignal 的核心产品价值 = **可信的世界杯数据分析**。数据不真实 → 产品价值为零。架构漂亮但数据假的 = 装修豪华的空房子。

3. **风险不对称**：架构重构引入回归 bug 的概率中等（代码审查明确指出「拆分 server.js 引入行为回归 bug」为中风险），修复这些 bug 消耗的时间会挤占数据修复的时间。而数据修复（ESPN roster、Odds API 替代）引入 bug 的概率低，因为它们主要是**新增或替换外部 API 调用**，不动现有路由逻辑。

4. **依赖关系**：数据是地基。架构重构不解决数据假的问题——架构方案中**没有任何一步涉及修复 ESPN roster、替换 Odds API、接通真实新闻源**。5 步全部是结构性的：抽 middleware、统一数据加载、拆路由、加版本号、前端 API 层。做完 5 步后，mock 数据问题**原封不动**。

### 2.2 「先修架构」的决定是否有充分理由？

让我客观分析「先修架构」的论点：

| 论点 | 评估 |
|------|------|
| 「架构乱了不好改数据」 | ⚠️ 部分成立。当前 server.js 的数据源调用（espn()、fetchJSON()）确实是内联的，但**它们已经能工作**。数据修复需要的是替换 ESPN roster 端点 URL 或接管 espn() 的返回值，不需要动路由结构。 |
| 「路由不匹配导致 API 假失效」 | ⚠️ 这是一个通讯问题，不是功能问题。35 个端点全部实际可用（代码审查已验证），API 假失效只是路径拼写问题。加一个 `/api` 索引端点（2-3h）和统一前端 API 层（4-5h）即可解决，**不需要走 5 步架构重构**。 |
| 「单体文件阻碍多人协作」 | ❌ 不成立。团队只有 **1 人**（全栈虾）。多人协作的模块化收益在当前阶段为零。 |
| 「先建好架构，数据接入更顺畅」 | ⚠️ 这是一个**时间换质量的权衡**。但考虑到世界杯赛程正在进行中，「时间」是这个项目最稀缺的资源。每多花一天在架构上，用户就多看一天假数据。 |

**结论：「先修架构」的论证不足以推翻 Sprint 规划师的建议。** 架构问题确实存在且需要解决，但它的紧迫性低于数据问题——尤其是在世界杯进行中的时间窗口下。

### 2.3 如果必须二选一，选哪个？

## 🔴 裁决：坚持 Sprint 规划师的「先修数据」路线

**理由汇总**：

```
数据真实 + 架构乱 → 产品有价值，可迭代
数据假   + 架构美 → 产品无价值，不可交付
```

但有一个**务实的折中空间**：

### 2.4 实际执行策略：窄架构 + 宽数据

| 阶段 | 做什么 | 时间 | 理由 |
|------|--------|------|------|
| **立即** | ✅ 完成已开始的 Step 1（infrastructure 提取） | 2h | 已经开始了就不要逆转，Step 1 风险低、不改变路由行为 |
| **然后** | ✅ 完成 Step 2（数据层统一，替代硬编码） | 1.5h | 这一步**本身就是数据修复的一部分**：消除硬编码 coach/venue，改用 JSON 文件 |
| **然后** | 🔴 **暂停架构重构**，全力修数据 | 5-6 天 | ESPN roster、Odds API、球员/球队真实统计、新闻源 |
| **然后** | ✅ 前端差异化功能 | 3-5 天 | SVG 对位图、角球追踪条、预测 Tab（此时已有真实数据） |
| **然后** | ✅ dataQuality 标记系统 | 1 天 | 🟢🟡🔴 可信度可视化 |
| **最后** | ⏸️ Step 3-5（路由拆分、版本化、前端 API 层） | 4-5 天 | 延期到 Sprint 2 |

**为什么这样安排**：

- Step 1 + Step 2 总共 3.5 小时，不会显著延迟数据修复。而且 Step 2 替代硬编码教练/场馆数据，本身就是数据质量提升。
- Step 3（路由拆分）是最高风险步骤（代码审查评级「中高」），在数据修复完成前做它，如果引入回归 bug，修复时间会挤占数据修复时间。
- Step 4（版本化）和 Step 5（前端 API 层）在真数据到位之前做，属于「在假数据上优化前端」——正是 Sprint 规划师警告要避免的。
